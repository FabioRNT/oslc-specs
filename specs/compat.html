This document will address considerations with backwards compatibility and co-existance considerations OSLC Core 2.0 when adopting 3.0. It is intended to cover the capabilities defined in OSLC Core 3.0.
This is an early working draft, working on outline and needs.

Introduction

The evolution of OSLC started from some simple documents that defined some XML and JSON formats and some interactions based on RESTful HTTP web services. These were labeled "OSLC 1.0". As a number of "1.0" specifications emerged, common patterns and reuse opportunities emerged. This then led to the creation of OSLC Core 2.0 [[OSLCCore2]]. From this, domain specifications were developed and other common specifications. The OSLC Core 2.0 evolved from 1.0 in that is strengthen its definition of the use of RESTful HTTP web services and adopted an existing standard-based approach to representing a data model that was flexible and had multiple representation formats, namely RDF [[rdf11-concepts]]. Now that OSLC Core 3.0 specifications are based on [[LINKED-DATA]] and namely the Linked Data Platform [[LDP]], there is a less-disruptive change from 2.0 to 3.0, as conceptually they are built on the same foundation: HTTP and RDF. Some of the underlying concepts and vocabularies may have changed in some areas, which this document will address how best to co-exist with 2.0 existing implementations. Which in the end, will mostly be able just producing a few more RDF triples (or statements) in some of the responses or additional HTTP response headers.

Motivation

There will be a period of time to allow for both the adoption of the 3.0 specification and the deployment of those clients and servers.

Changes from OSLC Core 2.0

The table below summarizes the capabilities from OSLC Core 2.0 based specifications and what changes have occurred within OSLC Core 3.0. It is not a complete list of all differences. As additional OSLC Core specifications are published, this table may be out of date and changes needed. The individual specifications and associated TCs will produce any needed backwards/forwards or cross dependency collateral.

Many capabilities are no longer specified in OSLC 3.0 because they are part of the W3C Linked Data Platform 1.0 [[LDP]]. Servers can continue to support both 2.0 and 3.0 capabilities for backwards compatibility with 2.0 clients.

Some capabilities are in both OSLC 2.0 and OSLC 3.0, but their implementation has changed. One example is Resource Preview [[OSLCPreview3]].

Note that the required resource representations have changed. In 2.0, these were often specialized at a domain level, such as RDF/XML and OSLC JSON representations may been required. Turtle [[turtle]] and JSON-LD [[JSON-LD]] are required instead, this is a [[LDP]] requirement.

Requirement Typical 2.0 Level 3.0 Level Comment
Unknown properties and content MAY / MUST MUST Unknown properties and content MUST NOT be ignored by servers or clients [[LDP]].
Resource Operations MUST MUST Resource operations are defined by [[LDP]].
Resource Paging MAY MAY Paging in OSLC 3.0 will use Linked Data Platform Paging 1.0 [[LDPPaging]]
Partial Resource Representations MAY / MUST Not specified oslc.properties is not specified in OSLC 3.0.
Partial Update MAY Not specified Partial update is not specified in OSLC 3.0. It will be replaced with the Linked Data Patch Format [[LDPatch]].
Service Provider Resources MAY / MUST Not specified Service provider resources are now simply discovery properties on LDP containers [[LDP]].
Creation Factories MUST Not specified Creation factories are now simply LDP containers [[LDP]], some additional properties are added to indicate which type and shapes are supported.
Query Capabilities MUST Not specified Query capabilities simply a way to filter LDP containers, using OSLC 2.0 Query Syntax [[LDP]].
Query Syntax MUST Not specified Query syntax is not specified in OSLC 3.0.
Delegated UI Dialogs MUST MAY Dialogs are no longer required. It's a capability servers MAY implement, however.
UI Preview SHOULD SHOULD The mechanics of resource previews have changed in 3.0 [[OSLCPreview3]]. The two versions can co-exist.
HTTP Basic Authentication MAY TBD Evaluating requirements for authentication in OSLC 3.0.
OAuth Authentication SHOULD TBD Evaluating requirements for authentication in OSLC 3.0.
Error Responses MAY MAY OSLC Services MAY provide error responses using Core defined error formats
RDF/XML Representations MUST / SHOULD Not specified XML and RDF/XML are not mentioned in OSLC 3.0. Servers can continue to support RDF/XML for backwards compatibility.
XML Representations MUST Not specified XML and RDF/XML are not mentioned in OSLC 3.0. Servers can continue to support XML for backwards compatibility.
OSLC 2.0 JSON Representations MUST Not specified The OSLC-specific JSON representation is not specified in OSLC 3.0. Instead, JSON-LD is REQUIRED in OSLC 3.0. Servers can continue to support the OSLC JSON format for backwards compatibility.
Turtle Representations Not specified MUST Turtle is REQUIRED [[LDP]].
JSON-LD Representations Not specified MUST JSON-LD is REQUIRED [[LDP]].
HTML Representations SHOULD SHOULD OSLC services SHOULD provide HTML representations for HTTP GET requests

OSLC-Core-Version

Deprecated. Technically, there is no to carry it forward in 3.0. Each capability should be able to accurately describe how to discover it using methods established by LDP such as HTTP response headers (Link relations, Accept-Post, ...).

Implementations that have previously tested for the value of this parameter, should ensure their logic properly accounts for values that may not be "2.0". As some may provide the value of "3.0", which shouldn't be seen as an error but instead as just an informative header.
